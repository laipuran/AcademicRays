# 📘 产品需求文档 (PRD) - Academic Rays

## 1. 产品概览

|**项目**|**内容**|
|---|---|
|**产品名称**|Academic Rays|
|**Slogan**|全链路智能学习伴侣：从预习到精通|
|**核心理念**|将“课前预习-课中记录-课后巩固”打通，利用 AI 实现知识的自动化归档、结构化整理与深度辅导。|
|**技术栈 (暂定)**|**客户端**: Kotlin Multiplatform (KMP) 或 Flutter<br><br>  <br><br>**后端**: Python (FastAPI/Django) + 向量数据库 (用于RAG) + LLM API|
|**目标用户**|K12学生、大学生、备考人群及教师|

## 2. 核心功能模块详解

基于“学习流”的时间轴进行功能梳理：

### 🟢 第一阶段：课前 (Pre-Class) —— 高效预习与规划

此阶段目标是**降低认知负荷，建立知识导图**。

#### 2.1 智能资料归档 (Smart Filing)

- **功能描述**：用户上传或拍摄即将学习的课件、PDF、参考书页。
    
- **AI 逻辑**：
    
    - 自动识别文件内容。
        
    - 将文件自动归类到对应的学科（如“高等数学”）和章节（如“微积分-极限”）。
        
    - _来源融合_：结合了上传文档中“文件的分类”功能。
        

#### 2.2 预习摘要与焦点提取 (Focus Extraction)

- **功能描述**：AI 分析课前材料，生成“预习简报”。
    
- **包含内容**：
    
    - **核心概念**：本节课将出现的关键术语（Key Terms）。
        
    - **前置知识检查**：提示用户“理解本节课内容，你最好先复习一下XX知识点”。
        
    - **预习清单**：根据材料难度和用户设定的时间（如“我有20分钟”），生成简短的阅读或思考任务流程。
        

---

### 🟡 第二阶段：课中 (In-Class) —— 自动化记录与结构化

此阶段目标是**通过多模态输入，实时捕捉并结构化知识**。

#### 2.3 课中快拍与实时归档 (Snap & Archive)

- **功能描述**：用户在课堂上拍摄板书、PPT。
    
- **AI 逻辑**：
    
    - **自动路由**：App根据当前时间表或GPS位置（可选），自动推断当前课程，直接将图片存入对应学科目录，无需手动选择。
        
    - **OCR 增强**：高精度识别手写体、印刷体及**复杂数学公式 (LaTeX格式)**。
        

#### 2.4 智能笔记生成 (Smart Notes)

- **功能描述**：基于图片OCR结果（及可选的录音片段），生成一份可编辑的Markdown笔记。
    
- **能力细节**：
    
    - **结构化重组**：将零散的板书图片转化为有层级（标题、正文、公式、代码块）的文档。
        
    - **遗漏补全**：(来自上传文档) AI 分析笔记的逻辑连贯性，如果发现板书跳跃，会自动标注“此处可能省略了推导步骤”，并尝试根据通用知识库补全或提示用户查阅。
        

#### 2.5 知识联想 (Knowledge Linking)

- **功能描述**：在生成的笔记中，通过**双向链接**或**悬浮卡片**展示关联知识。
    
- **场景**：当笔记中出现“泰勒展开”时，AI 自动链接到用户之前在“导数”章节的笔记，构建个人的知识图谱。
    

---

### 🔴 第三阶段：课后 (Post-Class) —— 闭环反馈与巩固

此阶段目标是**精准诊断，针对性提升**。

#### 2.6 智能批改与诊断 (AI Grading & Audit)

- **输入**：
    
    1. **题目信息**：用户拍摄/粘贴题干，或直接引用App内的题库。
        
    2. **作答内容**：用户拍摄手写的解题过程。
        
- **处理流程**：
    
    1. **识别**：OCR提取手写步骤。
        
    2. **逻辑比对**：AI 将用户步骤与标准解题路径进行比对。
        
    3. **反馈输出**：
        
        - **指出遗漏**：例如“在第3步到第4步之间，你漏掉了验根的步骤”。
            
        - **逻辑错误**：高亮显示错误推导的行。
            
        - **评分**：给出预估分数。
            

#### 2.7 动态任务流 (Dynamic Workflow)

- **待办集成**：批改后的错题自动生成“复习任务”，加入到用户的待办清单中。
    
- **个性化推题**：根据错题的知识点，AI 自动生成或检索 3-5 道**变式题**（Change-up problems），确保用户真的掌握了该逻辑，而不仅仅是记住了答案。
    

---

### 🟣 全局功能：Academic Ray (AI 助手)

- **Gemini 式辅导**：一个全局的 Chat 浮窗。
    
- **角色扮演**：用户可以设定它为“苏格拉底式导师”（只提问不给答案）或“直球式助教”（直接给解析）。
    
- **文档问答**：支持用户针对生成的笔记或上传的课件进行提问，“这一页PPT的核心论点是什么？”

## 3. 技术架构建议

### 3.1 客户端技术选型 (Flutter vs KMP)

鉴于你需要**拍照、扫描、手写渲染**以及未来的多端同步，以下是对比建议：

|**特性**|**Flutter**|**Kotlin Multiplatform (KMP)**|**推荐选择**|
|---|---|---|---|
|**UI 开发**|极快，自带渲染引擎，多端一致性完美。适合做复杂的自定义UI（如笔记编辑器、标注层）。|使用 Compose Multiplatform，目前在 iOS 端虽有进步但不如 Flutter 成熟。|**Flutter** (如果团队较小，追求UI快速落地)|
|**原生性能**|优秀，但在重度计算或调用原生相机API时需要写 Channel。|逻辑层共享，UI层原生。性能最接近原生，适合涉及大量底层交互。|**KMP** (如果未来打算深度利用Android/iOS的原生特性)|
|**生态库**|极其丰富 (pub.dev)。|正在快速增长，但部分第三方库可能不如Flutter全。|**Flutter**|

**建议**：若追求快速开发且UI交互复杂（如在图片上画圈批改），推荐使用 **Flutter**。

### 3.2 后端架构 (必须)

由于涉及 OCR、LLM 推理和数据存储，纯客户端无法承载，必须构建后端。

- **API Gateway**: 处理客户端请求，鉴权。
    
- **OCR 服务**: 接入 Google Cloud Vision API 或 Mathpix API (针对数学公式优化)。
    
- **LLM Service**: 封装 Gemini Pro 或 GPT-4o API，处理提示词工程 (Prompt Engineering)。
    
- **Database**:
    
    - **PostgreSQL**: 存储用户数据、结构化笔记、待办事项。
        
    - **Vector DB (如 Milvus/Pinecone)**: 存储笔记和题目的向量数据，用于实现“知识联想”和“RAG (检索增强生成)”。
        
- **Object Storage (AWS S3/OSS)**: 存储原始图片文件。

## 4. 用户交互流程 (User Flow) 示例：课后作业场景

1. **上传**: 学生打开 _Academic Rays_，点击“作业批改”，拍摄刚写完的物理试卷。
    
2. **输入题干**: 关联App内已有的电子版试卷，或直接拍摄题干。
    
3. **分析**: 屏幕显示“Rays正在扫描...”，后端进行 OCR 和 逻辑推理。
    
4. **反馈**:
    
    - 系统在图片上叠加红框，圈出第二行公式。
        
    - 弹窗提示：“此处动量守恒定律应用错误，忽略了摩擦力冲量。点击查看详细推导。”
        
5. **后续**:
    
    - 点击“加入错题本”。
        
    - 系统自动弹出一个 To-Do：“复习动量守恒 (明日截止)”。
        
    - Ray 助手弹出气泡：“需要我为你生成两道类似的含摩擦力动量题目吗？”
