# RFC #1: Academic Rays C/S 架构

* **状态**：Draft
* **作者**：DuckRan
* **日期**：2026-01-10
* **目标**：定义拍照归档流程中的前后端职责。

---

## 1. 架构决策：C/S 模式分配

为了保证移动端的轻量化以及 API 密钥（如 Mathpix, Gemini）的安全，我们采用 **Fat Server / Thin Client** 的策略。

### 1.1 客户端职责 (KMP Shared Module)

* **多模态采集**：负责调用系统相机拍摄板书、PPT 或作业，并进行简单的图片压缩处理。
* **本地缓存**：使用 **SQLDelight** 存储已同步的笔记索引，确保离线状态下可查看历史数据。
* **渲染层**：负责 Markdown 与 LaTeX 公式的混合渲染展示。

### 1.2 后端职责 (FastAPI + PostgreSQL)

* **归档引擎**：接收图片，调用 **OCR (Mathpix)** 提取文本，并驱动 **LLM (Gemini)** 进行学科分类与标题提取。
* **中心存储**：管理用户所有笔记、题目及对应的向量数据（用于未来的 RAG 检索）。
* **同步协调**：作为“单一真理来源”，处理多设备间的版本冲突。

---

## 2. 拍照归档核心流程 (Flow)

1. **Upload**: 客户端通过 Ktor 将图片流发送至 `/v1/archive/upload`。
2. **Processing**: 后端将图片推送到对象存储（如 S3/MinIO），同时异步发起 OCR 识别。
3. **Analysis**: 后端根据 OCR 文本调用 LLM 判断学科（数学/物理/等）并生成摘要。
4. **Response**: 返回结构化的 Markdown 内容及分类标签，客户端更新本地 SQLDelight。

---

## 3. 跨端同步机制 (Synchronization)

考虑到自用场景（手机拍照、平板学习），同步逻辑需遵循“简单有效”原则：

### 3.1 版本控制

* **Sequence ID**: 每条笔记拥有一个全局递增的 `version_tag`（由服务器维护）。
* **Sync Anchor**: 客户端记录本地已同步的最大 `version_tag`。

### 3.2 同步策略

* **增量拉取 (Pull)**：客户端启动或下拉刷新时，发送请求 `GET /sync?since={last_tag}`。
* **最后写入获胜 (LWW)**：若两端同时修改同一笔记，以服务器接收到的最后一次请求时间为准。

## 4. 讨论要点 (RFC Discussion)

针对你的队员，建议重点讨论以下三个问题：

1. **图片存储方案**：自用初期是直接存服务器磁盘，还是直接接入公有云 S3（考虑到后续扩展性，建议 S3）？
2. **OCR 实时性**：由于 Mathpix 和 Gemini 调用有延迟，客户端是否需要实现“归档中”的中间状态，而非阻塞式等待？
3. **流量优化**：KMP 端在上传前是否需要进行图片的 Base64 编码还是二进制流？（建议二进制流，节省约 30% 流量）。
